The container

A .lib is a book from a Library Tool archive sealed into a plain ZIP: everything an external tool — including an AI assistant with no prior knowledge of Library Tool — needs to understand, edit, and return it without breaking it. The file teaches its reader: every archive ships its own instructions and schema.

herbal.lib
├─ book.json            manifest: format version, identity, metadata,
│                       roles, stylesheet, templates, figures, page list
├─ INSTRUCTIONS.md      generated contract for humans and LLMs
├─ schema.json          JSON Schema (draft 2020-12) for the JSON members
├─ pages/<N>.json       one per page: dims, review state, regions
├─ translations/<lang>.json   page-aligned translations (optional)
└─ assets/img/<name>    figure crops referenced by figure regions

The current revision is format_version: "2.0" (“lib/2”). Versioning is MAJOR.MINOR: MINOR additions are optional and ignorable, a higher MAJOR is rejected with a clear error. Older lib/1 files (the bare "format": "lib/1" marker) import forever and upgrade on ingest.

The manifest (book.json)

keymeaning
format_version"2.0". Do not touch.
generatorThe writing tool, e.g. "library-tool/0.8.0".
book_idStable UUID, minted once per book and kept across re-exports — track a book across revisions with it. Do not touch.
metaBibliographic snapshot: title, subtitle, authors, year, publisher, edition, volume, language, pages, source_url.
capabilitiesWhat the file carries, e.g. ["norm-layer", "templates", "figures", "translations", "ext", "rid"].
rolesThe full region-role vocabulary as data: {role: {furniture, note}}. Furniture roles (running heads, margin notes, catchwords…) stay out of the compiled body text.
instructions{general_ref: "INSTRUCTIONS.md", book: "…"} — the per-book free-text guidance for editors and AI assistants.
stylesheetRole → modern-type mapping (family, size_em, style, variant, align, color, bg). Open-licensed faces only.
templatesReusable layout grids (geometry + roles, no text).
figuresFigure inventory: {name: {page, x, y, w, h, rework_of?}}, one entry per assets/img/ member.
pagesThe exported page numbers.
extThird-party namespace, round-tripped verbatim (size-capped). Also allowed per page and per region. The sanctioned home for custom data — everything else unknown is dropped.

Pages and regions

A page (pages/<N>.json) is a list of typed boxes in reading order:

{ "page": 7, "doc": "compiled.txt", "dims": { "w": 1400, "h": 1798, "dpi": 200 },
  "state": "verified",
  "items": [
    { "id": "r0", "rid": "k3f9a2", "role": "body", "src_type": "human",
      "order": 0, "box": { "x": 0.2, "y": 0.1, "w": 0.6, "h": 0.7 },
      "text": "diplomatic transcription…",
      "norm": "the modern reading…" } ] }
  • box — 0..1 fractions of the page (x, y top-left; w, h size). Never pixels.
  • text — the diplomatic transcription, faithful to the scan (long ſ, original spelling). norm — the optional modern-edition reading. Two layers of one region; keep both.
  • rid — stable identity, preserved across round trips; order — reading order; id — positional, regenerated on save (kept for back-compat).
  • A figure region's text is its ![id](id) placeholder — leave it be.

Editing rules

The invariants, stated imperatively. A file that follows them imports with zero warnings.

  1. Never renumber or rename pages/<N>.json — the page number is the key.
  2. Never invent roles; use the vocabulary in book.json.roles. Custom data goes in ext, never in new roles or new top-level keys.
  3. Translations and modernized text go in each region's norm layer or a translations/<lang>.json member — never overwrite text.
  4. Reworked or colorized images: write a new file under assets/img/ and add a figure entry with rework_of: "<original>" — never replace the original.
  5. Do not touch format_version, book_id, region rids, or the provenance fields (src_type, rework_of).

Translations

translations/<bcp47>.json carries page-aligned translated text. The app's translation store is page-grained, so each page's value is an object whose reserved _page key holds the whole page's text — a shape that stays valid if per-region (rid-keyed) translations arrive in a later MINOR revision. Import also accepts a plain string per page, or a rid → text map (joined in key order).

{ "lang": "ja", "pages": { "7": { "_page": "…翻訳されたテキスト…" } } }

Import receipt and validation

Nothing is dropped silently. Every import returns a receipt naming what landed and, in warnings[], everything that was coerced or skipped — with its location and reason. The same lint pass runs with no side effects against the app's local API (the desktop app serves it on loopback):

POST /api/lib/validate          multipart form, field "lib"

{ "ok": true, "format_version": "2.0", "pages": 12,
  "warnings": [ { "loc": "pages/3.json[1]",
                  "msg": "role 'headline' coerced to 'body': …" } ],
  "errors": [] }

With overwrite=1 on import, a figure whose name collides is replaced only when the incoming entry carries rework_of — deliberate rework wins, accidental collision skips with a warning.

Python API

tools/libformat.py is the format's single implementation — standalone, no web framework, importable by external scripts. The app's own import/export routes call the same code, so a file that round-trips through it cannot come out a shape the app rejects.

import libformat

doc = libformat.read_lib("herbal.lib")        # -> LibDocument
issues = libformat.validate(doc)              # -> [Issue(level, loc, msg)]
doc.pages[0].items[0]["norm"] = "the modern reading"
libformat.write_lib(doc, "herbal-edited.lib") # seals + re-sanitizes

libformat.ROLE_VOCAB          # role -> {furniture, note}
libformat.FORMAT_VERSION      # "2.0"
