# v0.8.0-alpha.16

Library Tool `0.8.0-alpha.16` is a desktop prerelease for hands-on testing.
Library Tool Capture is published separately as an Android product at
`0.5.1-alpha.10`; this desktop release does not rebuild or replace it.

## Highlights

- Corrections is now a real workbench. It opens as its own desktop window — the
  first independently launchable workbench, and the working proof of the
  KiCad-style manager direction described last release — for image cleanup,
  figure classification, and region-role review over captured pages, backed end
  to end by the framework-neutral engine rather than browser state.
- A perspective image editor lets you square up decoded page images directly in
  Corrections; decoded images route to it automatically even when they also
  carry region metadata.
- Image corrections run as durable background jobs. Queuing is idempotent: a
  retried or replayed request reuses the same operation receipt instead of
  starting duplicate work, and the corrected image commits atomically.
- Desktop capture intake seals each imported phone capture into a recoverable
  `.lib/3` archive linked by a portable, path-free association, and a captured
  book now carries a stable identity so retries and independent LAN or cloud
  delivery of the same capture converge on one book instead of minting
  duplicates.
- Desktop cloud sync reports progress one capture at a time, and desktop
  catalogue status — copyright and registration records, WHL and Internet
  Archive availability, scan status, remarks, and review state — now travels
  back to the phone over cloud and paired LAN.

## Changed

- Corrected images, figure categories, and region roles are read and written
  through revisioned engine contracts; category and region-role edits require an
  explicit `If-Artifact-Match` revision, so concurrent edits fail closed on a
  stale revision rather than overwriting each other. Correction transform workers
  are owned by the process, not the Corrections window, so a queued transform
  keeps running whether or not the workbench stays open.
- The desktop shell adds a workbench window registry (`desktop/main.js`,
  `desktop/window-registry.js`): an authenticated `workbench:open` path,
  per-workbench window identity and reuse, portable engine-context validation,
  and per-profile persistence of window bounds. The Tools menu gains **Open
  Corrections** and **Open New Corrections Window**; opening Corrections from a
  browser without the desktop app reports that it requires the desktop
  application.
- The cloud image processor and the desktop capture pipeline now share one
  framework-neutral raster kernel, so perspective correction and page-boundary
  geometry come from a single source revision. The packaged desktop sidecar
  bundles that kernel's runtime (`opencv-python-headless`, `numpy`), so
  perspective corrections work in the installed app and not only from a source
  checkout.
- Canonical metadata, taxonomy, or collection edits to a captured item mark its
  sealed archive `stale` through a recoverable transition before the catalogue
  commit, so an archive can never remain `current` after a successful edit while
  historical idempotency receipts stay verifiable. Capture identities are
  validated as portable, single-segment ids at every entry point, and capture
  ids colliding with reserved Windows device names are rejected.
- The `.lib/3` reader caps structural JSON depth with a string-aware scanner and
  validates `book.json` pages as a bounded, unique integer list that must match
  the physical `pages/<N>.json` members exactly; the libformat reader and the
  engine archive planner enforce the same boundary so their results cannot
  diverge.
- Capture ingest serializes cloud and paired-LAN delivery of the same capture,
  so one capture id can never publish two books or run two asset writers under a
  race, and Supabase capture-photo downloads enforce a 32 MB ceiling against both
  the advertised size and the streamed body.

## Fixed

- Corrections closes the Windows authority-swap race and concurrent authority
  writes, adds authority-ancestor verification on POSIX, rejects transforms whose
  command envelope does not match their route or that link to a foreign artifact
  revision, and validates that a transform commit produces a new, distinct
  artifact identity — so a correction can no longer overwrite or alias its own
  source image.
- A lost cloud acknowledgement now leaves the imported book durable and visible
  and leaves the remote row pending for an idempotent repair on the next run,
  instead of dropping the book or re-importing it. The sync run flag is claimed
  atomically, so a double press of **Sync** cannot start two concurrent runs.
- The phone review round-trip is crash-safe and revision-guarded: conflicting
  rows at the same revision conflict explicitly, a deleted and recreated review
  re-baselines instead of stranding a pending difference, and an exact no-op
  write no longer marks the sealed capture archive stale.
- Import refuses to proceed when a capture's derived book identity conflicts with
  a stored or linked identity, returning an explicit conflict instead of
  overwriting a stable id, and a captured item can no longer be detached or
  reassigned to a different capture on promotion.

## Architecture direction

The accepted direction remains a small project manager that opens independent
workbenches over the same library data; modules contribute capabilities,
commands, readiness, and views. Corrections is the first of those workbenches to
open in its own window. The engine has no Flask or Electron dependency, so a
later Qt client can embed it directly and a Godot client can use the same local
protocol. This release turns the engine, integrity, and packaging foundation
from alpha.15 into a first launchable workbench, not the final interface rebuild.

## Known caveats

- This remains a prerelease and should be tested on backed-up library data.
- The KiCad-style launcher is not complete: Corrections is the first
  independently launchable workbench, but the multi-workbench manager and
  separately packaged workbenches are not yet implemented — all modules still
  ship together.
- Importing a capture's page images into an **existing** catalogue item is not
  yet supported: a non-empty `.lib/3` capture graph is refused up front
  (`lib3_capture_graph_import_unsupported`) pending a raster/spatial persistence
  adapter. Sealing and promoting fresh captures is unaffected.
- Corrections figure re-OCR has no follow-up provider bound yet: a transform that
  requests re-OCR still commits the corrected image, then reports the follow-up
  as unavailable rather than producing new text.
- Windows installer signing depends on the configured release certificate. A
  privately managed certificate can still trigger SmartScreen on machines that
  do not trust its root.
