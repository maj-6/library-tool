# v0.8.0-alpha.14

Library Tool `0.8.0-alpha.14` and Android Book Capture `0.5.1-alpha.6`
are prerelease builds for hands-on testing.

## Highlights

- Book Capture now scans books into a **collection** — the batch a book
  belongs to, such as a shelf, a crate or a room — and records where that
  batch came from. Choosing a collection is required before a scan can start.
- Deleting a page, entry, manual record or translation now sends it to a
  recoverable **Trash** instead of destroying it, with a verbatim write-back
  restore.
- Book Capture's Home screen is one row of chrome instead of two: the app mark
  sits in a square beside the Scans and Collections tabs, and Settings, About
  and Check for updates hang off the mark.

## Changed

- A collection carries a **From** — "Storage", "Christopher Office" — that every
  book scanned into it inherits. A single book's origin can be changed
  afterwards from its detail screen, up until the capture uploads.
- Provenance is frozen per book when the scan starts, before the first photo, so
  changing collection halfway down a shelf never relabels books already
  captured, and a crash is recovered with the provenance the book began under.
- The collection name and the book's origin travel to the desktop inside the
  capture's metadata and appear in the entry's Extra section — no schema change
  and no new columns.
- Book Capture has a new launcher icon, framed so it survives circular,
  squircle and rounded-square launcher masks without clipping.
- Check for updates reads the published releases list and offers the newest
  build for the running channel. A stable build is never offered an alpha.
- Trash retains for 30 days, 200 items or 2 GiB, whichever comes first, and a
  restore surfaces where the restored item actually lives.

## Fixed

- A phone with no API key no longer files a blank entry. Scan provenance rides
  in the capture's metadata, which made every capture look as though the phone
  had already extracted the bibliography, so the desktop skipped its own OCR
  pass. This affected every LAN capture.
- Check for updates no longer reports "you have the newest build" when it could
  not check at all — a build with no cloud project configured now says so.
- Restoring from Trash advances the item's `updated_at`, so a restore is not
  silently overwritten by a stale sync.
- The blocked-mark guard cascades, and a thread revival is scoped to its own
  delete rather than reviving unrelated threads.

## Validation and infrastructure

- The local release gate passed Ruff, 653 Python tests, 123 JavaScript behavior
  tests, and 98 Android unit tests with lint clean.
- Book Capture was driven end to end on a booted API 34 emulator: the
  collection requirement holds on both the button and the spoken "start" route,
  the collection store survives a device restart, duplicate names are refused,
  and the app menu reaches Settings, About and update checks.
- `android/BookCapture/tools/emulator.ps1` boots a headless AVD with a bounded
  wait, so the app can be exercised rather than only built.
- `tools/make_android_icon.py` regenerates the launcher bitmaps from a single
  master and verifies the committed ones still match it.

## Known caveats

- This remains a prerelease.
- Collections live on the phone. They are not yet shared between devices or
  contributors, and there is no collections view on the desktop — a book's
  collection and origin appear in its entry's Extra section.
- A book's origin is fixed once its capture has uploaded, because the cloud
  capture row is insert-only.
- Update checks and the Downloads link open the browser rather than installing
  in place: these builds are not Play Store certified, so a self-install would
  land in the unknown-sources flow regardless.
- Windows installer signing depends on the configured release certificate. The
  current privately managed certificate may still trigger SmartScreen on
  machines that do not trust its root.
